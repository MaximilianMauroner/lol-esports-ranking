#### Anti-Patterns Verdict

This does not look like generic AI slop. It has a real data-product direction: compact dark analytics, strong domain terminology, and meaningful provenance. The problem is a mid-migration product surface, not a fake marketing shell.

The main tell is component vocabulary drift. shadcn/Radix primitives are present, but legacy CSS still owns much of the UI. Stock shadcn tokens, custom CSS variables, one-off inline spacing, native table CSS, div grids, and bespoke chart/table styles all coexist.

#### Overall Impression

The interface is credible and data-dense, but it lacks a single system for ranking rows, comparison surfaces, and controls. Desktop looks acceptable; mobile reveals the unfinished migration most clearly.

#### What's Working

- The top navigation and season scope create a stable mental model across Regions, Teams, and Players.
- The dark palette has a clear ranking/product feel, and the heat chips make score magnitude quickly scannable.
- The compare tray/drawer concept is useful: selected entities stay visible and removable.

#### Priority Issues

**[P1] Table/list systems are inconsistent across core views**

Why it matters: users learn one row pattern on Teams, another on Players, another on Regions, and another in Compare. On mobile this becomes scroll debt: Teams is 520px in a 333px wrapper, Players is 1040px in a 333px wrapper, and Compare is 438px in a 336px drawer table.

Fix: define one responsive ranking-row system. Use real tables when comparison by column matters, and a compact row-card layout on mobile when horizontal comparison is not viable. Make Regions either a real table/list with headings or a card list with explicit labels.

Suggested command: `impeccable adapt`.

**[P1] shadcn/Tailwind migration is visually functional but not normalized**

Why it matters: users see subtle mismatches in radius, active states, shadows, text color, and transitions. Engineers will keep fighting cascade order because shadcn stock classes and legacy app CSS both style the same primitives.

Fix: normalize Dialog, Sheet, Toggle, Tooltip, Skeleton, Button, Select, and Badge to app tokens. Remove duplicate legacy definitions once the primitive owns the pattern.

Suggested command: `impeccable extract`.

**[P2] Control semantics and affordances are mixed**

Why it matters: filters are announced as tabs, season tabs use pressed buttons, and selects hide native appearance without adding a chevron. This makes controls look similar while behaving differently.

Fix: use tabs only for actual view panels, segmented buttons for filters, and visible select chevrons. Keep active, hover, focus, and disabled state vocabulary identical across all controls.

Suggested command: `impeccable polish`.

**[P2] Mobile comparison flow depends on hidden horizontal scrolling**

Why it matters: the compare drawer opens on mobile, but the right column is clipped and discoverability depends on a tiny scrollbar. The core comparison task is weaker exactly where space is constrained.

Fix: stack compared entities by metric on mobile, or use a sticky metric column plus explicit horizontal-scroll affordance and enough width for both compared columns.

Suggested command: `impeccable adapt`.

**[P2] Accessibility polish is still incomplete**

Why it matters: custom row buttons and decorative swatches mostly work visually, but they deviate from semantic HTML. Screen reader and keyboard users get extra noise and less predictable interaction.

Fix: replace `role=button` rows with contained real buttons or links where possible, add `aria-hidden` to visual-only swatches, and keep sortable headers/control semantics standard.

Suggested command: `impeccable harden`.

#### Persona Red Flags

**Alex, power user:** Alex can scan rankings quickly on desktop, but mobile table overflow and non-URL-backed filters slow repeated analysis. The compare drawer is useful, but its table layout clips columns on phone width.

**Sam, accessibility-dependent user:** Sam encounters clickable `tr` and `div` rows with ARIA instead of native controls, empty visual `i` markers without `aria-hidden`, and filters exposed as tabs even when they are not panel tabs.

**Casey, distracted mobile user:** Casey sees the Players table become a 1040px horizontal scroll area, the compare drawer clip the second entity, and dense metadata at low contrast.

#### Minor Observations

- Season tabs are shadcn `Button` instances inside legacy `.season-tabs button` CSS.
- `.result-chip` is defined twice with different styling.
- shadcn `Skeleton` and legacy `.skeleton` both define loading visuals.
- `CountBadge` can look like a disabled input on mobile in the Teams toolbar.
- Several chart/profile swatches are empty `i` tags.

#### Questions to Consider

- Should mobile ranking views prioritize column comparison, or fast row scanning?
- Is the app committed to shadcn primitives as the source of truth, or is shadcn only a shell around legacy CSS?
- Should Regions use the same ranking table vocabulary as Teams, or should it intentionally become a card/list view?
